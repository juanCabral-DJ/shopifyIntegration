1. Arquitectura General 
FastAPI actúa como puerta de entrada para webhooks de Shopify y endpoints de sincronización o 
retry manual. Todo el trabajo pesado se delega a workers asíncronos usando Celery, RQ o ARQ, con 
Redis como broker y backend de resultados. 
PostgreSQL funciona como capa de idempotencia, trazabilidad y orquestación. La fuente de 
verdad transaccional permanece en el Sistema Externo, en adelante SE. El middleware no 
almacena datos fiscales ni transaccionales definitivos. 
1.1 Principio central  
- FastAPI no guarda ventas como sistema maestro. 
- Guarda eventos, mappings, snapshots y estados. 
- Todo lo fiscal, pagos, NCF , inventario final y facturación se confirma contra el SE. 
- Esto protege contra webhooks duplicados, caídas del SE, reintentos y diferencias de inventario. 
1.2 Componentes del sistema  
Componente Rol 
FastAPI Recibe webhooks Shopify, expone endpoints de 
sync y retry internos, valida HMAC. 
Celery / ARQ Workers asíncronos para insertar facturas, 
sincronizar productos, precios, inventario, 
imágenes y recibos. 
Redis Broker de tareas, caché de idempotencia y 
caché temporal de DTOs del SE. 
PostgreSQL Tablas de mapeo, event inbox/outbox, 
snapshots de inventario y caché durable de 
parámetros. 
Sistema Externo API REST. Fuente de verdad de productos, 
precios, stock, facturas y recibos. 
Shopify Canal de ventas. Emite webhooks. Recibe 
actualizaciones de inventario, precios e 
imágenes.

Flujos Clave 
5.1 Orden Shopify → Factura en el SE  
1. Shopify envía webhook `POST /webhooks/shopify/orders/create`. 
2. FastAPI valida HMAC-SHA256. 
3. Guarda payload crudo en `event_inbox`. 
4. Verifica idempotencia en `map_order_ids`. 
5. Encola job `insert_invoice_job`. 
6. Worker resuelve vendedor, NCF , cliente, condición de pago y sucursal. 
7. Transforma la orden a `MfactrxDto`. 
8. Llama `POST /api/Factura/Insertar`. 
9. Guarda relación en `map_order_ids` y `map_invoices`. 
10. Si falla, reintenta con backoff exponencial. 
 
 5.2 Pagos no -Shopify  
 
1. Worker `payment_polling_job` corre periódicamente. 
2. Consulta recibos recientes en el SE. 
3. Verifica si `eftrcb_numero` existe en `map_recibos`. 
4. Actualiza `financial_status = paid` en Shopify. 
5. Registra recibo y auditoría. 
6. Si Shopify falla, reintenta hasta 5 veces. 
 
 5.3 Sincronización de inventario  
 
1. Worker consulta `POST /api/Minvitm/get`. 
2. Complementa con `GET /api/Minvfismovil/getfisico`. 
3. Compara stock SE contra stock Shopify. 
4. Guarda snapshot. 


--- PAGE 9 ---
5. Ajusta Shopify si hay diferencia. 
6. Marca `reconciled = true`. 
7. Si hay discrepancy persistente, emite alerta. 
8. Al confirmar fulfillment, aplica ajuste al SE vía `POST /api/inventario/fisico`.

8. Seguridad y Resiliencia 
8.1 Seguridad  
Control Implementación 
HMAC-SHA256 Validación de cada webhook Shopify antes de encolar. 
TLS 1.2+ Toda comunicación externa por HTTPS. 
Secrets Credenciales en AWS Secrets Manager o HashiCorp Vault. 
API Key / OAuth SE Token cacheado en Redis con TTL. 
Privacidad Datos sensibles no se loguean en texto plano. 
 
 
 


--- PAGE 16 ---
8.2 Resiliencia  
Patrón Detalle 
Idempotencia factrx_movil_id, shopify_order_id y eftrcb_numero con constraints únicos. 
Backoff exponencial Reintentos con delay progresivo hasta 5 intentos. 
Circuit breaker Cliente HTTP hacia SE con tenacity o lógica equivalente. 
DLQ Eventos muertos pasan a cola dead-letter. 
Retención Eventos retenidos hasta 24h si el SE está caído. 
Timeout SE Timeout recomendado de 30 segundos.
